VOLUME CONCEPT (used in Jenkinsfile):
  args '-v pip-cache:/root/.cache/pip --user root'

  What is a Docker volume?
    - A volume is persistent storage that survives container restarts/removal.
    - Containers are ephemeral (temporary) — when removed, all data inside is lost.
    - Volumes let you keep data between container runs.

  Why use it here?
    - Without volume: pip downloads Flask, pytest etc EVERY build (~10 sec each time).
    - With volume: pip cache is saved. Next build reuses cached packages. Much faster.

  Types of volumes (interview):
    - Named volume:   docker run -v my-vol:/data         → Docker manages storage location
    - Bind mount:     docker run -v /host/path:/data     → You control exact host path
    - tmpfs mount:    docker run --tmpfs /data            → In-memory only, lost on stop

  Interview Q: "What is a Docker volume?"
    "A volume is persistent storage managed by Docker that outlives containers.
     We use it to cache build dependencies between CI runs so pip/npm/maven
     don't re-download everything on each build."

  Why '--user root' was added:
    - Jenkins started Docker agent as non-root user (uid 1000).
    - pip failed with permission denied while installing packages.
    - '--user root' lets pip install to system site-packages and write cache.
    - This is acceptable in CI for short-lived build containers.

  Production note:
    - Avoid root where possible.
    - Better approach: build a custom CI agent image with dependencies pre-installed,
      or use controlled writable paths for non-root users.

Jenkins job configuration (for reference):
  - Job type: Pipeline
  - Definition: Pipeline script from SCM
  - SCM: Git
  - Repository URL: https://github.com/adityaverma4444/ci-cd-app.git
  - Branch: */main
  - Script Path: flask-cicd-app/Jenkinsfile
  - Credentials: none (public repo)

Key Jenkins concepts (interview):
  - Declarative vs Scripted pipeline:
      Declarative = pipeline { } block, structured, easier to read (we use this)
      Scripted = node { } block, full Groovy, more flexible but harder

  - agent any vs agent none vs agent docker:
      any    = run on Jenkins master or any connected node
      none   = don't assign globally, each stage picks its own
      docker = spin up a container from given image for that stage

  - Fail-fast behavior:
      If any stage fails, remaining stages are SKIPPED.
      Post block still runs (for notifications/cleanup).
      Exit code 127 = "command not found" in Linux.

  - BUILD_NUMBER:
      Auto-incremented by Jenkins (1, 2, 3...).
      Used to tag Docker images for traceability.

  - checkout scm:
      Pulls code from the SCM (Git) configured in the Jenkins job.

                    STAGE 4: GITHUB WEBHOOK (NEXT)
================================================================================

Goal:
  Auto-trigger Jenkins build whenever code is pushed to GitHub.

Why this stage matters:
  - Removes manual "Build Now"
  - Makes CI event-driven
  - This is the real-world CI trigger pattern

Pre-checks:
  - Jenkins job already works with manual Build Now (done)
  - GitHub repo is reachable from Jenkins (done)
  - Jenkins has URL accessible from GitHub

Webhook setup (GitHub):
  1) Open repo -> Settings -> Webhooks -> Add webhook
  2) Payload URL:
       http://<your-jenkins-host>:8080/github-webhook/
     For local machine testing, expose Jenkins with ngrok:
       ngrok http 8080
     Then use:
       https://<ngrok-id>.ngrok-free.app/github-webhook/
  3) Content type: application/json
  4) Which events: Just the push event
  5) Active: checked
  6) Save webhook

Jenkins job setup:
  - Job -> Configure -> Build Triggers
  - Enable: GitHub hook trigger for GITScm polling
  - Save

Validation test:
  1) Make a small commit (e.g., comment in notes or README)
  2) git push origin main
  3) Check Jenkins build queue/history
  4) Build should start automatically without clicking Build Now

Common webhook issues:
  - 404/timeout from GitHub:
      Jenkins not publicly reachable (use ngrok/localtunnel)
  - 403 No valid crumb:
      Usually wrong endpoint path; must end with /github-webhook/
  - No build trigger:
      Build trigger checkbox not enabled in Jenkins job
  - Secret mismatch:
      If using secret, configure same secret on both GitHub and Jenkins

Stage 4 execution log:
  - Used ngrok to expose Jenkins: https://discriminatingly-unequivalent-karma.ngrok-free.dev
  - Added webhook in GitHub: Payload URL = <ngrok-url>/github-webhook/
  - Enabled "GitHub hook trigger for GITScm polling" in Jenkins job config
  - GitHub Recent Deliveries showed:
      ping  → green checkmark (200) at setup time
      push  → green checkmark (200) after git push
  - Jenkins build #9 triggered automatically after push
  - Build #9 shows "No Changes" because Jenkinsfile was already up to date on that push
  - Build #8 was the last successful full pipeline run (all stages green)

  "No Changes" explained:
    - Webhook fired correctly (GitHub delivered push event, Jenkins received it).
    - Jenkins compared latest commit with last built commit.
    - If no NEW code changes since last build, stages still run but show "No Changes".
    - Build #9 failed likely due to Docker Build stage (same as before), not webhook.
    - Webhook itself worked perfectly.

  Q: "If webhook has no changes, will it still run?"
    - YES, webhook triggers the build regardless.
    - Jenkins receives the push event and starts the pipeline.
    - "No Changes" just means no new commits since last build.
    - The pipeline still executes all stages.
    - This is different from SCM Polling, which checks for changes first and
      only triggers if there are new commits.

  How to verify webhook worked:
    1) GitHub → Webhooks → Recent Deliveries → green checkmark = GitHub sent it
    2) Jenkins → Build History → new build appears = Jenkins received it
    3) Console Output → first line says "Started by GitHub push by <username>"
       (vs "Started by user unknown" for manual Build Now)

deployment.yaml explained:
  apiVersion: apps/v1        → K8s API group for Deployments
  kind: Deployment           → Resource type that manages pod lifecycle
  replicas: 2                → Run 2 pod copies (high availability)
  selector.matchLabels       → How Deployment finds its Pods (label matching)
  template                   → Blueprint for each Pod
  image: flask-cicd-app:latest → Which Docker image to run
  imagePullPolicy: Never     → Use local image, don't pull from registry
  containerPort: 5000        → Port app listens on inside container

service.yaml explained:
  kind: Service              → Stable network endpoint for Pods
  type: NodePort             → Exposes on node IP + fixed port (30000-32767)
  selector: app: flask-cicd  → Routes traffic to pods with this label
  port: 5000                 → Service listens on this (cluster-internal)
  targetPort: 5000           → Forwards to this port on Pod
  nodePort: 30080            → External port on Minikube node

  Traffic flow:
    Browser → minikube-ip:30080 → Service:5000 → Pod:5000 → Flask app

  Label matching chain (interview critical):
    Deployment selector → finds Pods with label app=flask-cicd
    Service selector    → finds Pods with label app=flask-cicd
    Both point to same pods. This is how traffic routes correctly.

Service types (interview must-know):
  ClusterIP     → Internal only, no external access (default)
  NodePort      → Exposes on static port on each node (we used this)
  LoadBalancer  → Creates cloud LB on AWS/GCP/Azure (production)

Interview answers from Stage 5:
  Q: "What is a Pod?"
    "Smallest deployable unit in K8s. Wraps one or more containers that
     share network and storage."

  Q: "What is a Deployment?"
    "Manages Pods — ensures desired replicas are running, handles rolling
     updates, auto-restarts failed pods."

  Q: "What is a Service?"

  Q: "ClusterIP vs NodePort vs LoadBalancer?"
    "ClusterIP is internal-only. NodePort exposes on a static port on
     each node. LoadBalancer creates a cloud LB in production."

YAML STRUCTURE GUIDE (interview must-know)
--------------------------------------------------------------------------------

  Deployment YAML structure (indentation matters):

    apiVersion / kind / metadata     ← top-level resource info
    spec:                            ← Deployment spec
      replicas                       ← how many pods
      selector                       ← how Deployment finds its Pods
      template:                      ← Pod blueprint
        metadata:
          labels                     ← Pod labels (must match selector)
        spec:                        ← Pod spec level
          containers:                ← list of containers
            - name / image / ports   ← container basics
              livenessProbe          ← inside container (per-container)
              readinessProbe         ← inside container (per-container)
              resources              ← inside container (per-container)
              volumeMounts           ← inside container (WHERE to mount)
          volumes:                   ← pod level (WHAT to mount, shared)
          tolerations:               ← pod level (scheduling rules)

  Rule of thumb:
    - Things specific to ONE container → inside containers list
    - Things shared across pod / scheduling → same level as containers

--------------------------------------------------------------------------------
1. LIVENESS & READINESS PROBES
--------------------------------------------------------------------------------

  Added to deployment.yaml inside container spec:

    livenessProbe:
      httpGet:
        path: /health
        port: 5000
      initialDelaySeconds: 10
      periodSeconds: 15

    readinessProbe:
      httpGet:
        path: /health
        port: 5000
      initialDelaySeconds: 5
      periodSeconds: 10

  Liveness probe:
    - "Is this container alive?"
    - If fails → K8s RESTARTS the container
    - Catches: app hung, deadlocked, zombie process

  Readiness probe:
    - "Is this container ready to serve traffic?"
    - If fails → K8s REMOVES pod from Service (no traffic sent)
    - Does NOT restart. Just waits.
    - Catches: app still starting, warming cache, loading config

  Why both?
    - Liveness handles crashes/hangs (restart)
    - Readiness handles slow starts (hold traffic)
    - Together = zero-downtime deployments

  Probe types (interview):
    httpGet   → HTTP request to endpoint (we used this)
    tcpSocket → TCP connection check
    exec      → Run a command inside container

  Interview Q: "Liveness vs Readiness probe?"
    "Liveness restarts unhealthy containers. Readiness stops traffic to
     pods not yet ready. Both are needed: liveness for self-healing,
     readiness for zero-downtime deployments."

--------------------------------------------------------------------------------
2. RESOURCE REQUESTS & LIMITS
--------------------------------------------------------------------------------

  Added to deployment.yaml inside container spec:

    resources:
      requests:
        cpu: "100m"
        memory: "128Mi"
      limits:
        cpu: "250m"
        memory: "256Mi"

  Requests:
    - Minimum guaranteed resources
    - K8s scheduler uses this to decide which node to place pod on
    - 100m = 0.1 CPU core (10% of one core)
    - 128Mi = 128 megabytes

  Limits:
    - Maximum allowed resources
    - CPU exceeded → throttled (slowed down)
    - Memory exceeded → OOMKilled (pod killed and restarted)
    - 250m = 0.25 CPU core
    - 256Mi = 256 megabytes

  Why set both?
    - Without requests: scheduler can't plan, pods may land on overloaded nodes
    - Without limits: one pod can eat all node resources, starving others

  Interview Q: "What happens if a pod exceeds its memory limit?"
    "It gets OOMKilled — Kubernetes kills it and restarts based on
     restartPolicy. This prevents one pod from consuming all node memory."

--------------------------------------------------------------------------------
3. PERSISTENT VOLUME (PV) & PERSISTENT VOLUME CLAIM (PVC)
--------------------------------------------------------------------------------

  Created k8s/pv-pvc.yaml:

    PV (PersistentVolume):
      - Cluster-level storage resource (like a hard disk)
      - Provisioned by admin or dynamically
      - capacity: 100Mi, accessModes: ReadWriteOnce
      - hostPath: /data/flask-app (local path on node, dev only)

    PVC (PersistentVolumeClaim):
      - Pod's request for storage
      - Specifies how much and what access mode
      - K8s binds PVC to a matching PV automatically

  Mounted in deployment.yaml:
    volumeMounts (inside container):
      - name: app-data
        mountPath: /app/data     ← path INSIDE container where data appears

    volumes (pod spec level):
      - name: app-data
        persistentVolumeClaim:
          claimName: flask-pvc   ← links to PVC created in pv-pvc.yaml

  Flow:
    PV created → PVC created → K8s binds them → Pod mounts PVC → /app/data works

  Access modes (interview):
    ReadWriteOnce (RWO)  → One node reads/writes
    ReadOnlyMany (ROX)   → Many nodes read only
    ReadWriteMany (RWX)  → Many nodes read/write

  hostPath vs cloud storage:
    hostPath  → local node directory (dev/testing only, data lost if node dies)
    EBS/EFS   → AWS persistent block/file storage
    Azure Disk → Azure managed persistent disk
    GCE PD    → Google persistent disk

  Interview Q: "What is PV/PVC?"
    "PV is cluster-level storage, PVC is a pod's request for that storage.
     K8s binds them together. I used hostPath for local dev; in production
     you'd use cloud-backed storage like EBS or managed disks."

--------------------------------------------------------------------------------
4. TAINTS & TOLERATIONS
--------------------------------------------------------------------------------

  Added to deployment.yaml at pod spec level:

    tolerations:
      - key: "env"
        operator: "Equal"
        value: "production"
        effect: "NoSchedule"

  Taint (on node):
    - "No Entry" sign on a node
    - Command: kubectl taint nodes minikube env=production:NoSchedule
    - Blocks pods from scheduling on this node

  Toleration (on pod):
    - "Special pass" that lets pod bypass the taint
    - Pod says "I can handle env=production"

  Without toleration → pod blocked from tainted node
  With toleration    → pod can schedule on tainted node

  Taint effects (interview):
    NoSchedule        → New pods blocked, existing pods stay
    PreferNoSchedule  → K8s tries to avoid, not guaranteed
    NoExecute         → New pods blocked AND existing pods evicted

  Real-world use cases:
    - GPU nodes: taint gpu=true → only ML pods with toleration go there
    - Production nodes: taint env=prod → only prod apps
    - Control-plane: already tainted so user pods don't run there

  Commands:
    kubectl taint nodes minikube env=production:NoSchedule      ← add taint
    kubectl taint nodes minikube env=production:NoSchedule-     ← remove taint

  Interview Q: "What are taints and tolerations?"
    "Taints are on nodes to repel pods; tolerations are on pods to allow
     scheduling on tainted nodes. We use them to dedicate nodes for specific
     workloads like GPU or production environments."

5. SCALING
--------------------------------------------------------------------------------

  Manual scaling:
    kubectl scale deployment flask-cicd-app --replicas=4
    kubectl get pods      ← shows 4 pods
    kubectl scale deployment flask-cicd-app --replicas=2
    kubectl get pods      ← back to 2 pods

  HPA (Horizontal Pod Autoscaler):
    Prerequisite: minikube addons enable metrics-server
    kubectl autoscale deployment flask-cicd-app --cpu-percent=50 --min=2 --max=5
    kubectl get hpa

    How it works:
      - Watches average CPU usage across all pods
      - If CPU > 50% target → adds pods (up to max=5)
      - If CPU drops → removes pods (down to min=2)
      - Requires resource requests to be set (we did this)

  Interview Q: "How does HPA work?"
    "HPA monitors CPU/memory metrics via metrics-server. When average usage
     exceeds the target threshold, it scales up replicas. When usage drops,
     it scales down. It requires resource requests to calculate percentages."

6. ROLLING UPDATE (bonus — happened during Stage 6A apply)
--------------------------------------------------------------------------------

  What triggered it:
    - We changed deployment.yaml (added probes, resources, volume, tolerations)
    - kubectl apply detected pod template changed
    - K8s started replacing old pods with new ones gradually

  What we saw:
    flask-cicd-app-7cf86cf4c8-lqbfz   1/1  Running   ← old pod
    flask-cicd-app-7cf86cf4c8-w22g5   1/1  Running   ← old pod
    flask-cicd-app-7df54c8955-hm6qg   0/1  Running   ← new pod (readiness pending)

  How rolling update works:
    1) Create new pod with updated config
    2) Wait for readiness probe to pass (0/1 → 1/1)
    3) Route traffic to new pod
    4) Kill one old pod
    5) Repeat until all old pods replaced

  Where is it defined?
    - It is the DEFAULT strategy for Deployments (you don't need to write it)
    - Explicit version:
        spec:
          strategy:
            type: RollingUpdate
            rollingUpdate:
              maxSurge: 1           ← max 1 extra pod during update
              maxUnavailable: 0     ← never fewer than desired count

  Rolling Update vs Recreate:
    RollingUpdate → gradual replacement, zero downtime (default)
    Recreate      → kill all old pods first, then create new = DOWNTIME

  When does rolling update trigger?
    - Image changed          → yes
    - Probes/resources added → yes
    - Labels/env changed     → yes
    - replicas changed       → NO (that's just scaling, existing pods stay)

  Interview Q: "What is a rolling update?"
    "Default deployment strategy in K8s. When pod template changes, K8s
     gradually replaces old pods with new ones — creating first, waiting
     for readiness, then terminating old. Ensures zero downtime."

WHY JENKINS CAN'T REACH MINIKUBE LOCALLY (important to understand)
--------------------------------------------------------------------------------

  Your setup:
    Jenkins   → runs inside Docker container (my-jenkins)
    Minikube  → runs inside Docker container (minikube)
    kubeconfig → points to 127.0.0.1:1793

  Problem:
    From inside Jenkins container, 127.0.0.1 = Jenkins itself, NOT Minikube.
    They are isolated Docker containers with separate network stacks.
    kubectl from Jenkins can't reach Minikube API server.

  This is a LOCAL Docker-in-Docker networking limitation, NOT a CI/CD design flaw.

  HOW WE FIXED IT (full automation achieved):
    Step 1: Connected Jenkins to Minikube's Docker network
      docker network connect minikube my-jenkins
      Why: Jenkins was on "jenkins" network, Minikube on "minikube" network.
           They couldn't see each other. Now Jenkins has IP 192.168.49.3 on
           the minikube network and can reach Minikube at 192.168.49.2.

    Step 2: Installed kubectl inside Jenkins container
      docker exec --user root my-jenkins bash -c "curl -kLO <kubectl-url> && mv kubectl /usr/local/bin/"
      Why: Jenkins didn't have kubectl binary to run K8s commands.

    Step 3: Copied kubeconfig + TLS certs into Jenkins
      docker cp ~/.minikube/ca.crt my-jenkins:/var/jenkins_home/.minikube/ca.crt
      docker cp ~/.minikube/profiles/minikube/client.crt my-jenkins:/var/jenkins_home/.minikube/profiles/minikube/client.crt
      docker cp ~/.minikube/profiles/minikube/client.key my-jenkins:/var/jenkins_home/.minikube/profiles/minikube/client.key
      Why: kubectl needs TLS certificates to authenticate with K8s API.

    Step 4: Created kubeconfig inside Jenkins pointing to Minikube internal IP
      server: https://192.168.49.2:8443 (NOT 127.0.0.1)
      Why: 127.0.0.1 from inside Jenkins = Jenkins itself. 192.168.49.2 = Minikube.

    Step 5: Verified from inside Jenkins
      docker exec my-jenkins kubectl get nodes → minikube Ready
      docker exec my-jenkins kubectl get deployment → flask-cicd-app 2/2

  Result: Jenkins can now run kubectl commands against Minikube.
          Deploy stage in Jenkinsfile works end-to-end.

  In production this networking issue doesn't exist because:
    - Jenkins and K8s cluster share the same VPC/network
    - Or Jenkins uses a K8s service account token + cluster API URL
    - Or GitOps tools (ArgoCD/FluxCD) handle deployment separately

  NOTE FOR FUTURE CHATS:
    If Docker Desktop restarts, Minikube may need restart (minikube start).
    Jenkins kubectl setup persists inside the container but network connect
    may need to be re-run: docker network connect minikube my-jenkins
    Also re-check Minikube IP hasn't changed: docker inspect minikube

STAGE 6C: IMAGE TRANSFER — DinD TO MINIKUBE (THE TRICKY PART)
================================================================================

PROBLEM: TWO SEPARATE DOCKER DAEMONS
-------------------------------------
  Jenkins (via jenkins-docker DinD) and Minikube each run their own Docker
  daemon. Images built by Jenkins live in DinD's daemon. Minikube's K8s
  needs images in Minikube's Docker daemon. They can't see each other.

  Symptom: kubectl set image succeeds, but pods show "ErrImageNeverPull"
           because Minikube's Docker doesn't have the image.

  Architecture:
    Docker Desktop (Host)
      ├── my-jenkins      → Jenkins master, has kubectl
      ├── jenkins-docker  → DinD, Docker daemon #2 (images built HERE)
      └── minikube        → Docker daemon #3 (images needed HERE)

SOLUTION: IMAGE-LOADER POD
---------------------------
  Created k8s/image-loader.yaml — a temporary pod with Docker socket mounted:

    volumes:
      - name: docker-sock
        hostPath:
          path: /var/run/docker.sock   ← Minikube's Docker socket

  /var/run/docker.sock is the API doorway to Minikube's Docker daemon.
  Mounting it gives the pod direct access to run Docker commands against
  Minikube's Docker.

DEPLOY STAGE STEPS (in Jenkinsfile):
--------------------------------------
  1. docker save flask-cicd-app:build-N -o /tmp/flask-image.tar
     → Exports image from DinD as a tar file onto Jenkins master's disk

  2. kubectl delete pod image-loader --ignore-not-found
     → Clean up any leftover pod from previous builds

  3. kubectl apply -f k8s/image-loader.yaml
     → Creates image-loader pod (docker:cli image + Docker socket mount)

  4. kubectl wait --for=condition=Ready pod/image-loader --timeout=120s
     → Waits until pod is running and ready

  5. kubectl cp /tmp/flask-image.tar image-loader:/tmp/flask-image.tar
     → Copies the tar from Jenkins into the pod via K8s API

  6. kubectl exec image-loader -- docker load -i /tmp/flask-image.tar
     → Loads image into MINIKUBE's Docker (via mounted socket)
     → Now Minikube has the image!

  7. kubectl delete pod image-loader --ignore-not-found
     → Cleanup the temporary pod

  8. rm -f /tmp/flask-image.tar
     → Cleanup the tar file

  9. kubectl set image deployment/flask-cicd-app flask-cicd=flask-cicd-app:build-N
     → Updates the deployment to use the new image (rolling update starts)

  10. kubectl rollout status deployment/flask-cicd-app --timeout=120s
      → Waits for rolling update to complete (new pods healthy)

  11. kubectl get pods -l app=flask-cicd
      → Shows final pod status as confirmation

KEY CONCEPTS:
--------------
  Docker socket (/var/run/docker.sock):
    - Unix socket file that Docker daemon listens on
    - Any process with access to this file can control Docker
    - Mounting it into a container = giving that container Docker access
    - Security risk in production (container can control the host's Docker)

  docker save / docker load:
    - save: exports image as a .tar archive (all layers + metadata)
    - load: imports a .tar archive back into a Docker daemon
    - Used for offline image transfer or air-gapped environments

  kubectl cp:
    - Copies files between local machine and a pod
    - Uses tar under the hood (pod must have tar binary)
    - Goes through the K8s API server (slower than direct copy)

  hostPath volume:
    - Mounts a file/directory from the HOST NODE into a pod
    - NOT portable (depends on what's on the specific node)
    - Used here to access Minikube's Docker socket from inside a pod

WHY 3.5 MINUTES?
-----------------
  docker save  → packaging ~200MB image → ~15-20s
  kubectl cp   → transferring tar via K8s API → ~60-90s (slowest part)
  docker load  → unpacking layers → ~20-30s
  rollout      → starting pods + health checks → ~30-60s

IN PRODUCTION:
--------------
  You would NOT do this. Instead:
    Jenkins builds → docker push to DockerHub/ECR/GCR → K8s pulls from registry
  The image-loader is a local workaround for isolated Docker daemons.

ERRORS ENCOUNTERED AND FIXED:
-------------------------------
  Error 1: bitnami/kubectl Docker agent — "localhost:8080 refused"
    Cause: Deploy stage used a Docker agent (bitnami/kubectl) which created
           a NEW container without kubeconfig. kubectl defaulted to localhost:8080.
    Fix: Changed to agent any (runs on Jenkins master where kubectl is configured)

  Error 2: ErrImageNeverPull — rollout timed out
    Cause: Image built in DinD, not in Minikube's Docker. imagePullPolicy: Never
           means Minikube won't try a registry, just looks locally.
    Fix: Added image-loader pod to transfer image from DinD → Minikube's Docker

INTERVIEW Q: "How did you handle image transfer in your local CI/CD?"
  "Jenkins and Minikube ran in separate Docker containers with isolated

What was completed in 6A:

What remained after 6A:

Core 6B problem (most important):
  - jenkins-docker (DinD) and minikube use different Docker daemons.
  - Jenkins build creates image in DinD daemon.
  - Kubernetes needs image in Minikube daemon.
  - They do not share image storage.
  - Symptom: ErrImageNeverPull / rollout timeout.

Why DinD created image transfer problem:
  - Image built by Jenkins -> stored in jenkins-docker Docker.
  - Image required by K8s -> must exist in minikube Docker.
  - No shared image cache between these daemons in local setup.

Bridge solution used:
  - Temporary image-loader pod + mounted Docker socket.
  - docker save (Jenkins side) -> kubectl cp -> docker load (Minikube side).
  - Then kubectl set image + rollout status.
  - This made deployment fully automated in local environment.

Extra 6B issue fixed:
  - Deploy stage initially used bitnami/kubectl docker agent.
  - That container had no kubeconfig and defaulted to localhost:8080.
  - Fix: use agent any for Deploy and run kubectl on Jenkins container where kubeconfig exists.

STAGE 7 — HELM MIGRATION (Kubernetes Package Manager)
================================================================================

What is Helm?
  - Helm is like apt/yum but for Kubernetes.
  - Without Helm: you write multiple YAML files and apply them one by one.
  - With Helm: you write templates with {{ placeholders }}, put all config in
    one values.yaml, and deploy everything with a single command.
  - Interview one-liner: "Helm is a Kubernetes package manager that packages
    manifests into reusable charts with templated values."

Key Helm Concepts:
  - Chart     = a folder/package containing templates + config
  - Templates = K8s YAML files with {{ .Values.xxx }} placeholders
  - values.yaml = single file with all configurable parameters
  - Release   = a running instance of a chart (has revision history)
  - _helpers.tpl = reusable Go template functions for names and labels

What we did:

  Step 1: Created chart skeleton
    Command: helm create flask-cicd-app
    Generated folder: flask-cicd-app/helm/flask-cicd-app/
    Structure:
      Chart.yaml          -- metadata (chart name, version)
      values.yaml         -- all configurable values
      templates/
        deployment.yaml   -- K8s Deployment with {{ placeholders }}
        service.yaml      -- K8s Service with {{ placeholders }}
        _helpers.tpl      -- reusable name/label generators
        pvc.yaml          -- we added this ourselves for PersistentVolumeClaim
        NOTES.txt         -- printed after helm install
        tests/            -- helm test connection pod

  Step 2: Configured values.yaml
    This is where Helm's power lives. Instead of editing 3-4 YAML files,
    you edit ONE file. We set:
      replicaCount: 2
      image.repository: flask-cicd-app
      image.pullPolicy: Never
      image.tag: latest
      service.type: NodePort
      service.port: 5000
      resources (requests/limits for CPU and memory)
      livenessProbe and readinessProbe (path: /health)
      volumes (PVC: flask-pvc-helm2)
      volumeMounts (mountPath: /app/data)
      tolerations (key: env, value: production)

  Step 3: How templates pull values (the core concept)
    In templates/deployment.yaml:
      image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
    Helm replaces this with:
      image: "flask-cicd-app:latest"
    Same for replicas:
      replicas: {{ .Values.replicaCount }}  -->  replicas: 2
    The template is the skeleton, values.yaml is the brain.

  Step 4: Created templates/pvc.yaml
    Helm didn't generate this by default. We added it to create the
    PersistentVolumeClaim via Helm. Named flask-pvc-helm2 to avoid
    conflict with old manually-created flask-pvc.

  Step 5: Set fullnameOverride and serviceAccount.create
    fullnameOverride: flask-cicd-helm2
      Controls the name prefix for all resources Helm creates.
    serviceAccount.create: false
      We had deleted the serviceaccount.yaml template earlier.
      Helm's deployment template referenced a SA that didn't exist.
      Setting create: false makes pods use the "default" SA.
      This fixed: "serviceaccount flask-cicd-helm2-flask-cicd-app not found"

  Step 6: Deployed to helm-lab namespace
    kubectl create namespace helm-lab
    Used separate namespace to keep Helm deployment isolated from
    old kubectl-applied deployment in default namespace.

Commands used and what each does:

  1) helm template (dry run / preview)
     helm template flask-cicd-helm2 helm/flask-cicd-app -n helm-lab \
       -f helm/flask-cicd-app/values.yaml
     - Renders all templates locally, prints final YAML to terminal.
     - Nothing is sent to Kubernetes. It's a preview.
     - We used this to verify the YAML looks correct before deploying.

  2) helm upgrade --install (deploy)
     helm upgrade --install flask-cicd-helm2 helm/flask-cicd-app -n helm-lab \
       -f helm/flask-cicd-app/values.yaml
     - If release does NOT exist -> install it (first time).
     - If release already exists -> upgrade it (update values/templates).
     - This single command replaces kubectl apply -f for all files.
     - In CI/CD, always use upgrade --install for idempotency.

  3) helm list -n helm-lab
     - Shows all Helm releases in the namespace.
     - Output: release name, namespace, revision, status, chart version.

  4) helm uninstall flask-cicd-helm2 -n helm-lab
     - Deletes ALL resources the chart created in one command.
     - Replaces running kubectl delete for each resource separately.

  5) helm rollback flask-cicd-helm2 <revision>
     - Rolls back to a previous revision.
     - Helm keeps history of all revisions.
     - View history with: helm history flask-cicd-helm2

Error we hit and fix:

  Error: PersistentVolumeClaim "flask-pvc" exists and cannot be imported
         invalid ownership metadata; missing key "app.kubernetes.io/managed-by"
  Cause: In stages 5-6 we manually created flask-pvc via kubectl apply.
         That PVC had no Helm labels. Helm refuses to adopt existing
         resources it didn't create.
  Fix:   Changed PVC name to flask-pvc-helm2 so Helm creates its own PVC.
  Interview point: "Helm tracks ownership via labels
    (app.kubernetes.io/managed-by: Helm) and annotations
    (meta.helm.sh/release-name). Pre-existing resources without these
    labels cause ownership conflicts."

  Error: serviceaccount "flask-cicd-helm2-flask-cicd-app" not found
  Cause: Deployment template referenced a ServiceAccount that didn't exist
         because the SA template file was deleted, but values.yaml still
         had serviceAccount.create: true (due to duplicate keys).
  Fix:   Set serviceAccount.create: false so pods use default SA.

_helpers.tpl explained:

Interview cheat sheet for Helm:

  Q: What is Helm?
  A: "Helm is a package manager for Kubernetes. It packages K8s manifests
     into reusable charts with templated values."

  Q: What is a Helm Chart?
  A: "A chart is a collection of files: Chart.yaml (metadata),
     values.yaml (config), and templates/ (K8s YAMLs with Go template
     syntax). It's a blueprint for deploying an application."

  Q: What is a Helm Release?
  A: "A release is a running instance of a chart. Same chart can be
     installed multiple times with different release names. Each release
     tracks its own revision history."

  Q: helm install vs helm upgrade vs helm upgrade --install?
  A: "Install creates new release, upgrade updates existing one,
     upgrade --install does both. In CI/CD we always use upgrade --install
     for idempotency."

  Q: How do you rollback?
  A: "helm rollback <release> <revision>. Helm keeps revision history.
     View with helm history <release>."

  Q: What is values.yaml?
  A: "Single source of truth for all configurable parameters. Templates
     reference values using {{ .Values.xxx }}. Override at install time
     with --set or -f flags."

  Q: How does Helm track resource ownership?
  A: "Helm adds labels (app.kubernetes.io/managed-by: Helm) and
     annotations (meta.helm.sh/release-name) to every resource.
     This is how it knows which resources belong to which release."

Important difference between health check and DB check:

    - Good for liveness/readiness probes.
    - Does NOT prove DB connectivity.

    - Proves service name resolution works (DB_HOST=mariadb-lab).
    - Proves DB credentials/env vars reached the container.
    - Proves SQL query execution works.

Meaning of successful verification:
  If /health works:

  If /db-check works:

Application code added for DB connectivity:

    - reads DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD from env vars

Helm integration for DB connectivity:

    DB_HOST=mariadb-lab
    DB_PORT=3306

    DB_PASSWORD=app1234

MariaDB service explanation:

    healthy but still-initializing database.

  Important concept:

Three ways to access MariaDB manually:

1) From inside the MariaDB pod

  Enter the pod:

    SHOW TABLES;

2) From local machine using port-forward

    kubectl port-forward -n helm-lab service/mariadb-lab 3307:3306

  Connect from local client:
    mariadb -h 127.0.0.1 -P 3307 -u appuser -p appdb

  Good for:
    - running SQL from host machine
    - using DB GUI/client tools

    Our current Flask image contains Python connector library, but not the
    mariadb/mysql CLI client binary.

    kubectl exec -it -n helm-lab deploy/flask-cicd-helm2 -- sh

    python -c "import mysql.connector; c=mysql.connector.connect(host='mariadb-lab',port=3306,database='appdb',user='appuser',password='app1234'); print('connected')"

  This is actually closer to real application behavior.

How to get mysql/mariadb CLI inside Flask pod (future improvement):
  If we want interactive SQL from inside Flask pod, we must add a DB client to
  the Flask image.

  Since our image is based on python:3.12-slim, typical Dockerfile addition is:
    RUN apt-get update && apt-get install -y default-mysql-client && rm -rf /var/lib/apt/lists/*

  After rebuilding and redeploying, inside Flask pod we can run:
    kubectl exec -it -n helm-lab deploy/flask-cicd-helm2 -- sh
    mariadb -h mariadb-lab -P 3306 -u appuser -p appdb

About ".mysql" or client config inside the pod:
  In Linux, the usual file is ~/.my.cnf, not ".mysql".
  It can store default client connection settings for convenience.

  Example ~/.my.cnf:
    [client]
    host=mariadb-lab
    port=3306
    user=appuser
    password=app1234
    database=appdb

  Then the CLI can be used without repeating all flags:
    mariadb

  Important interview note:
    Storing plain password in ~/.my.cnf is convenient for lab/testing, but in
    real projects we should prefer Kubernetes Secrets or mounted secret files.

Goal:
  Replace the old kubectl-based deploy in Jenkins with Helm.
  Jenkins pipeline builds the image, then runs helm upgrade --install
  to deploy to Kubernetes using the Helm chart.

What changed:
  - Jenkinsfile (main branch):
      The old "Deploy to K8s" stage (kubectl set image) was replaced with
      a "Deploy with Helm" stage:
        helm upgrade --install flask-cicd-helm2 helm/flask-cicd-app \
          -n helm-lab \
          --set image.tag=build-${BUILD_NUMBER} \
          --wait --timeout 120s
      This tells Helm to install (or upgrade if exists) the release named
      flask-cicd-helm2 using the chart at helm/flask-cicd-app, in the
      helm-lab namespace.

  - Multibranch Pipeline (dev branch):
      The dev branch keeps the OLD kubectl-based deploy style for learning.
      It uses kubectl apply for manifests, docker save/load for image transfer,
      and kubectl set image for rolling updates. This demonstrates how teams
      can have different deploy strategies per branch.

  - Helm was manually installed inside the Jenkins container:
      curl -kLO https://get.helm.sh/helm-v3.14.4-linux-amd64.tar.gz
      tar -zxvf helm-v3.14.4-linux-amd64.tar.gz
      mv linux-amd64/helm /usr/local/bin/helm
    Helm is NOT on the persistent volume, so if the Jenkins container is
    recreated, Helm must be reinstalled.

Key concepts:
  helm upgrade --install:
    - If the release doesn't exist → installs it (like helm install)
    - If the release exists → upgrades it (like helm upgrade)
    - This is idempotent — safe to run multiple times

  --set flags:
    Override values.yaml at deploy time without editing the file.
    Example: --set image.tag=build-5 overrides image.tag in values.yaml

  --wait:
    Helm waits until all pods are Running/Ready before declaring success.
    Without this, Helm would exit immediately even if pods are crashing.

  --timeout 120s:
    If pods aren't ready within 120 seconds, Helm rolls back and fails.

Interview points:
  Q: Why use Helm in a Jenkins pipeline instead of kubectl?
  A: Helm manages releases as a unit — you can rollback, version, and template.
     kubectl apply is imperative and doesn't track release history.

  Q: What does helm upgrade --install do?
  A: It's idempotent. Installs if new, upgrades if exists. Safe for CI/CD
     because you don't need to check if the release exists first.

Green flag for Stage 9:
  - Jenkins pipeline runs helm upgrade --install successfully
  - Pods come up healthy in helm-lab namespace
  - dev branch still works with old kubectl apply style

Jenkins Credentials Setup (dockerhub-creds):
  1. Go to Jenkins → Manage Jenkins → Credentials → Global
  2. Add Credentials → Kind: Username with password
  3. ID: dockerhub-creds
  4. Username: your Docker Hub username
  5. Password: your Docker Hub Personal Access Token (PAT)
     (NOT your Docker Hub login password — use a PAT with read/write scope)

Docker Hub PAT (Personal Access Token):
  - Created at: hub.docker.com → Account Settings → Security → Personal access tokens
  - Scopes: repo:read,write (for push) or repo:read (for pull only)
  - You CANNOT retrieve a PAT after creation — save it when generated
  - If lost, create a new one and update Jenkins credentials

Kubernetes imagePullSecrets — why needed:
  Problem:
    Jenkins (inside my-jenkins container) has Docker Hub credentials.
    But Kubernetes (inside minikube) does NOT. They are separate systems.
    When a pod tries to pull an image from Docker Hub, it pulls anonymously.
    Docker Hub rate-limits anonymous pulls → "toomanyrequests" error.

  Solution:
    Create a Kubernetes secret that stores Docker Hub credentials:
      kubectl create secret docker-registry dockerhub-pull \
        -n helm-lab \
        --docker-server=https://index.docker.io/v1/ \
        --docker-username=adityaverma268 \
        --docker-password=<YOUR_PAT>

    Then tell the pod to use it via imagePullSecrets in the deployment spec.
    We pass this through Helm: --set imagePullSecrets[0].name=dockerhub-pull

  Important:
    - This secret lives inside the Kubernetes cluster
    - If minikube is DELETED and recreated, the secret is gone → recreate it
    - If minikube is just RESTARTED, the secret survives

STAGE 11 — Kubernetes Secrets (Creation, Consumption, CI/CD Integration)
==========================================================================

What are Kubernetes Secrets?
  - Kubernetes objects that store sensitive data (passwords, tokens, keys)
  - Stored in etcd (the cluster's key-value store)
  - Base64-encoded by default — NOT encrypted (base64 is encoding, not security)
  - To encrypt at rest, enable EncryptionConfiguration on the API server
  - Accessed by pods via env vars or volume mounts

Secret types we used:
  1. Opaque (generic) — for arbitrary key-value data (DB credentials)
  2. kubernetes.io/dockerconfigjson — for container registry auth (imagePullSecrets from Stage 10)

ConfigMap vs Secret (interview favourite):
  ┌──────────────────┬──────────────────────┬──────────────────────┐
  │                  │     ConfigMap        │     Secret           │
  ├──────────────────┼──────────────────────┼──────────────────────┤
  │ Purpose          │ Non-sensitive config  │ Sensitive data       │
  │ Stored as        │ Plain text           │ Base64-encoded       │
  │ Example          │ DB_HOST, DB_PORT     │ DB_PASSWORD, tokens  │
  │ Mounted as       │ Env var or file      │ Env var or file      │
  │ RBAC             │ Same                 │ Can be restricted    │
  │ Encryption       │ No                   │ Optional at-rest     │
  └──────────────────┴──────────────────────┴──────────────────────┘

-----------------------------------------------
TWO WAYS TO CONSUME SECRETS IN PODS
-----------------------------------------------

1. As Environment Variable (secretKeyRef):
   In values.yaml:
     env:
       - name: DB_PASSWORD
         valueFrom:
           secretKeyRef:
             name: db-file-secret
             key: DB_PASSWORD

   Verification:
     kubectl exec <pod> -n helm-lab -- printenv DB_PASSWORD
     → app1234

   Use when: App reads config from env vars (12-factor app pattern, most common)

2. As Volume Mount (projected as files):
   In values.yaml:
     volumes:
       - name: db-secrets
         secret:
           secretName: db-file-secret
     volumeMounts:
       - name: db-secrets
         mountPath: /etc/db-secrets
         readOnly: true

   Each key becomes a file: /etc/db-secrets/DB_PASSWORD contains "app1234"

   Verification:
     kubectl exec <pod> -n helm-lab -- ls /etc/db-secrets/
     → DB_HOST  DB_NAME  DB_PASSWORD  DB_PORT  DB_USER
     kubectl exec <pod> -n helm-lab -- cat /etc/db-secrets/DB_PASSWORD
     → app1234

   Use when:
     - App expects config files (e.g. TLS certs, .my.cnf, SSH keys)
     - You need auto-rotation (volume mounts update when secret changes, env vars don't)

   Key difference: Volume-mounted secrets auto-update when the Secret object changes.
   Env var secrets require a pod restart to pick up new values.

-----------------------------------------------
WITHCREDENTIALS SYNTAX EXPLAINED
-----------------------------------------------

  withCredentials([usernamePassword(
      credentialsId: 'db-creds',        ← ID from Jenkins > Manage Credentials
      usernameVariable: 'DB_USER',      ← mapped to $DB_USER in shell
      passwordVariable: 'DB_PASS')]) {  ← mapped to $DB_PASS, masked in logs
    sh "helm upgrade ... --set db.user=$DB_USER --set db.password=$DB_PASS"
  }

  - credentialsId: Must match the ID you gave when adding credentials in Jenkins UI
  - usernameVariable/passwordVariable: Temp env vars, only exist inside the {} block
  - Jenkins auto-masks these values in console output (shows ****)
  - Supports other types too: string(), file(), sshUserPrivateKey()

PRODUCTION BEST PRACTICES (interview gold)
-----------------------------------------------

  1. Never commit real secrets to Git — use dummy/default values in values.yaml
  2. Jenkins credential store for CI-time secrets → passed via --set at deploy time
  3. For production clusters, use external secrets managers:
     - HashiCorp Vault + Vault Agent Injector
     - Sealed Secrets (Bitnami) — encrypts secrets so they're safe in Git
     - External Secrets Operator — syncs from AWS Secrets Manager / Azure Key Vault
  4. Enable encryption at rest for etcd (EncryptionConfiguration)
  5. Use RBAC to restrict who can read secrets:
     kubectl create role secret-reader --verb=get --resource=secrets
  6. Rotate secrets regularly — volume mounts auto-update, env vars need pod restart

INTERVIEW Q&A — STAGE 11
-----------------------------------------------

  Q: How do you manage secrets in Kubernetes?
  A: We use Kubernetes Secrets with stringData for creation. Secrets are consumed
     via secretKeyRef (env vars) or volume mounts (files). For CI/CD, Jenkins stores
     credentials in its credential store, injects them via withCredentials, and passes
     to Helm --set flags so real values never touch Git.

  Q: What's the difference between secretKeyRef and volume mount for secrets?
  A: secretKeyRef injects a single key as an env var — simple but requires pod
     restart on secret change. Volume mount projects all keys as files under a
     directory — supports auto-rotation and is better for certs/config files.

  Q: stringData vs data in a Secret manifest?
  A: stringData accepts plain text, Kubernetes auto-encodes to base64.
     data requires you to pre-encode values. stringData is easier to write;
     both produce the same Secret object.

  Q: How do you handle secrets in CI/CD?
  A: Jenkins stores credentials in its credential store, accessed via
     withCredentials in the pipeline. At deploy time, credentials are passed
     as Helm --set overrides, so the Secret template uses real values while
     Git only has dummy defaults. Never commit secrets to Git.

  Q: What if someone does kubectl get secret -o yaml?
  A: They see base64-encoded values — which can be trivially decoded.
     That's why RBAC matters: restrict secret read access. For true security,
     enable etcd encryption at rest and use external secrets managers.

  Q: How would you rotate a database password in production?
  A: Update the Secret object (kubectl edit or through the secrets manager).
     Volume-mounted secrets auto-update in ~1 minute. Env-var-based secrets
     require a pod restart (kubectl rollout restart deployment/xxx).

  Q: Why not just use ConfigMaps for everything?
  A: ConfigMaps are plain text and visible to anyone with read access.
     Secrets get base64 encoding, optional encryption at rest, and separate
     RBAC controls. Use ConfigMaps for non-sensitive config (ports, hostnames)
     and Secrets for passwords, tokens, and keys.

  Q: What is Sealed Secrets?
  A: A Bitnami project. You encrypt secrets locally with a public key
     (kubeseal CLI) → produces a SealedSecret resource safe to commit to Git.
     The controller in-cluster decrypts it using its private key and creates
     a regular Kubernetes Secret. Solves the "secrets in Git" problem.

--------------------------------------------------------------------------------
1. WHY MONITORING?
--------------------------------------------------------------------------------
  - Without monitoring, debugging production issues is guesswork
  - You can't SSH into every server to check logs manually at scale
  - Monitoring gives real-time visibility: request rate, latency, error rates
  - You open a dashboard and immediately see what's wrong and when it started

  Interview: "Monitoring gives us visibility into application health and

--------------------------------------------------------------------------------
2. THE THREE PLAYERS (Restaurant Analogy)
--------------------------------------------------------------------------------
  Flask App (app.py)  = Kitchen that keeps a tally board of orders
  Prometheus          = Manager who reads the tally board every 15 seconds
  Grafana             = TV screen that shows pretty charts from the manager's notes

  Flow: Flask counts requests → Prometheus collects every 15s → Grafana draws graphs

  Key concept: Prometheus uses a PULL model. It visits your app's /metrics endpoint.
  Your app does NOT push data to Prometheus. This is different from push-based
  systems like Datadog agent.

--------------------------------------------------------------------------------
3. WHAT WE CHANGED IN CODE
--------------------------------------------------------------------------------

  FILE 1: app.py — Added 4 things:

    a) Two metric objects (lines 9-18):
       REQUEST_COUNT = Counter('flask_http_requests_total', ...)
         → Counts total requests. Labels: method, endpoint, status
         → Counter ONLY goes up (never decreases)
       REQUEST_LATENCY = Histogram('flask_http_request_duration_seconds', ...)
         → Records how long each request takes
         → Histogram gives you percentiles (p50, p95, p99)

    b) before_request hook (lines 21-23):
       → Starts a stopwatch before every request

    c) after_request hook (lines 26-39):
       → Stops the stopwatch
       → Adds +1 to REQUEST_COUNT for this method/endpoint/status
       → Records the latency in REQUEST_LATENCY
       → Skips recording for /metrics itself (avoids infinite loop)

    d) /metrics endpoint (lines 76-78):
       → Returns all metric data in Prometheus text format
       → This is what Prometheus visits every 15 seconds
       → Output looks like:
         flask_http_requests_total{method="GET",endpoint="/",status="200"} 150
         flask_http_request_duration_seconds_bucket{endpoint="/",le="0.005"} 148

  FILE 2: requirements.txt — Added prometheus-client library

  Counter vs Histogram:
    Counter  = "How many?" (only goes up, e.g., total requests)
    Histogram = "How long?" (distribution of values, gives percentiles)
    You'd never use a Counter for latency because you need the range, not just total.

--------------------------------------------------------------------------------
4. SERVICEMONITOR — The Bridge Between App and Prometheus
--------------------------------------------------------------------------------

  File: helm/flask-cicd-app/templates/servicemonitor.yaml

  In plain English: "Hey Prometheus, there's an app in helm-lab namespace
  with label flask-cicd-app. Hit its /metrics every 15 seconds."

  Key fields:
    release: prometheus     → Permission tag. Prometheus only scrapes
                              ServiceMonitors with this label.
    namespaceSelector:
      matchNames: [helm-lab] → Look in helm-lab namespace
    selector:
      matchLabels:
        app.kubernetes.io/name: flask-cicd-app → Find this Service
    endpoints:
      port: http, path: /metrics, interval: 15s → Scrape config

  The chain:
    ServiceMonitor → finds Service "flask-cicd-app" in helm-lab
    → Service points to Flask pods → Prometheus scrapes /metrics every 15s

  Why not edit Prometheus config directly?
    ServiceMonitors are Kubernetes-native (declarative YAML, version-controlled,
    deployed via Helm). Manual config editing doesn't scale and isn't GitOps-friendly.

  What is a CRD (Custom Resource Definition)?
    ServiceMonitor is NOT a built-in K8s resource. The Prometheus Operator
    installs it as a CRD. It extends Kubernetes with new resource types.

7. GRAFANA DASHBOARD — The 3 PromQL Queries
--------------------------------------------------------------------------------

  Panel 1: Request Rate
    Query: rate(flask_http_requests_total[5m])
    Meaning: "How many requests per second over last 5 minutes?"

  Panel 2: Request Latency (p95)
    Query: histogram_quantile(0.95, rate(flask_http_request_duration_seconds_bucket[5m]))
    Meaning: "95% of requests completed within how many seconds?"

  Panel 3: Total Requests by Endpoint
    Query: sum by (endpoint, status) (flask_http_requests_total)
    Meaning: "Total count of requests grouped by endpoint and status code"

  What is PromQL?
    Prometheus Query Language. How you ask questions about your metrics data.
    rate()               → per-second rate over a time window
    sum by (label)       → group and aggregate
    histogram_quantile() → calculate percentiles from histogram data

  Dashboard = the whole page. Panel = one graph/chart inside the dashboard.
  We created 1 dashboard ("Flask App Monitoring") with 3 panels.

10. INTERVIEW Q&A
--------------------------------------------------------------------------------

  Q: How do you monitor your applications?
  A: We use Prometheus and Grafana. The app exposes a /metrics endpoint using
     the prometheus_client Python library. Prometheus scrapes it every 15 seconds
     via a ServiceMonitor CRD. Grafana visualizes the data with dashboards
     showing request rate, latency percentiles, and error rates.

  Q: What metrics do you track?
  A: Request count by method/endpoint/status (Counter), and request latency
     (Histogram). Counter tells us throughput and error rates. Histogram gives
     percentile latency like p95 and p99.

  Q: What's the difference between Counter and Histogram?
  A: Counter only goes up — it counts total events. Histogram records the
     distribution of values (like latency) so you can calculate percentiles.

  Q: What is a ServiceMonitor?
  A: A Kubernetes CRD created by the Prometheus Operator. Instead of manually
     editing Prometheus config to add scrape targets, you create a ServiceMonitor
     YAML that says "scrape this service, on this port, at this path." The
     Operator watches for these and auto-configures Prometheus.

  Q: What's the difference between Prometheus and Grafana?
  A: Prometheus collects and stores metrics (the database). Grafana is the
     visualization layer that queries Prometheus and draws dashboards.

  Q: How does Prometheus discover your app?
  A: Pull-based model. Prometheus pulls data from /metrics. ServiceMonitor
     tells it where to pull from. Different from push-based systems like
     Datadog where the app sends data to the collector.

  Q: What is PromQL?
  A: Prometheus Query Language. Example: rate(flask_http_requests_total[5m])
     means "per-second rate of requests over last 5 minutes." Supports
     rate(), sum(), histogram_quantile() for aggregating time-series data.

  Q: What are the three pillars of observability?
  A: Metrics (Prometheus/Grafana), Logs (EFK stack/Loki), Tracing (Jaeger/
     OpenTelemetry). We implemented metrics in this stage.

  Q: What would you alert on?
  A: Error rate > 5% (too many 500s), p95 latency > 2 seconds, pod restart
     count increasing, and request rate dropping to zero (app might be down).

4. PORT BASICS (the part that was confusing)
--------------------------------------------------------------------------------

  What is a port?
    A door number on a machine. Each app picks a door to stand behind.
    Flask = door 5000. MariaDB = door 3306. NGINX = door 80.

  "Listening" = an app is waiting behind that door for visitors.

  Three Service Types:
    ClusterIP    → No external door. Internal only. (MariaDB: 3306)
    NodePort     → Random external door (32678) on your machine. (Flask old way)
    LoadBalancer → External door 80 with nice address. (Ingress Controller)

  5000:32678 means: door 32678 on your machine → forwards to port 5000 internally

  What is path: / ?
    URL = host + path.  http://flask.local/health
                         ^^^^^^^^^^^        ^^^^^^
                         HOST               PATH
    path: / means "match everything" (/, /health, /metrics, /db-check)

              app.kubernetes.io/instance=flask-cicd-helm2

  Deployment gives these same labels to every pod it creates.
  Service scans: "who has these labels?" → finds 2 pods → adds them as Endpoints.

  ENDPOINTS = the actual pod IPs the Service discovered.
  kubectl get endpoints shows: 10.1.0.134:5000, 10.1.0.137:5000
  kubectl get pods -o wide shows: same IPs. They match.

  Service = fixed phone number. Endpoints = the actual desks it rings.

8. INTERVIEW Q&A
--------------------------------------------------------------------------------

  Q: What is Ingress?
  A: A K8s resource that manages external HTTP access to services. It provides
     URL-based routing, SSL termination, and load balancing through one entry
     point (port 80/443) instead of multiple NodePorts.

  Q: What's the difference between Ingress and a Service?
  A: Service connects pods internally and provides a stable endpoint.
     Ingress sits in FRONT of the Service and handles external HTTP routing
     by hostname and path. Ingress needs a Service to forward to.

  Q: What is an Ingress Controller?
  A: The actual software (like NGINX) that reads Ingress rules and routes
     traffic. Without a controller running, Ingress resources do nothing.
     Common controllers: NGINX, Traefik, HAProxy, AWS ALB, GCE.

  Q: Why not just use NodePort?
  A: NodePort wastes ports (one random port per service), doesn't support
     host/path-based routing, and can't do SSL termination. Ingress gives
     one clean entry point for all services.

  Q: What's the hosts file for?
  A: Maps domain names to IPs locally. Used in dev because flask.local isn't
     a real domain. In production, real DNS (Route53, CloudFlare) handles this.

  Q: How would you route multiple services with Ingress?
  A: Multiple rules in the Ingress resource:
       flask.local/         → flask-service:5000
       flask.local/api      → api-service:8080
       admin.local/         → admin-service:3000
     One NGINX controller handles all of them.

  Q: NodePort vs ClusterIP vs LoadBalancer?
  A: ClusterIP = internal only (database). NodePort = random external port.
     LoadBalancer = cloud-provided external IP on port 80/443.

                    STAGE 14: ALERTING (PrometheusRule + Alertmanager)
================================================================================

WHAT THIS STAGE DOES:
  Prometheus already collects metrics (Stage 12). Now we define RULES that
  say "if this condition stays true for X minutes, fire an alert."
  Alertmanager receives fired alerts and can send email, Slack, PagerDuty.

--------------------------------------------------------------------------------
1. THE THREE PLAYERS
--------------------------------------------------------------------------------

  Prometheus       → Collects metrics, evaluates alert rules every 30s
  PrometheusRule   → CRD (Custom Resource) that defines the alert condition
  Alertmanager     → Receives fired alerts, routes to email/Slack/PagerDuty

  Flow:
    Rule condition true → PENDING → (for: 1m) → FIRING → Alertmanager → Notify

--------------------------------------------------------------------------------
2. ALERT STATES
--------------------------------------------------------------------------------

  INACTIVE  = Rule is loaded. Condition is false. All good.
  PENDING   = Condition just became true. Waiting for "for: 1m" timer.
  FIRING    = Condition was true for full duration. Alert is active.

  Why "for: 1m" exists: prevents false alarms from brief 30-second blips.
  Only alert if the problem persists for a full minute.

--------------------------------------------------------------------------------
3. THE PROMETHEUSRULE CRD
--------------------------------------------------------------------------------

  File: helm/flask-cicd-app/templates/prometheusrule.yaml

  {{- if .Values.alertRules.enabled }}
  apiVersion: monitoring.coreos.com/v1
  kind: PrometheusRule                    ← NOT a built-in K8s object
  metadata:
    name: flask-cicd-helm2-alerts
    namespace: helm-lab
    labels:
      release: kube-prom                 ← CRITICAL label (see section 4)
  spec:
    groups:
      - name: flask.rules
        rules:
          - alert: FlaskAppDown
            expr: absent(up{namespace="helm-lab"}) or up{namespace="helm-lab"} == 0
            for: 1m
            labels:
              severity: critical
            annotations:
              summary: "Flask app is down"
  {{- end }}

  Controlled by values.yaml:
    alertRules:
      enabled: true

  Jenkins deploys it with:
    --set alertRules.enabled=true

--------------------------------------------------------------------------------
4. THE release: kube-prom LABEL -- WHY IT MATTERS
--------------------------------------------------------------------------------

  Prometheus Operator scans the cluster for PrometheusRule objects.
  But it does NOT load ALL rules it finds. It filters by label:

    kubectl get prometheus -n monitoring -o jsonpath="{.items[0].spec.ruleSelector}"
    → {"matchLabels":{"release":"kube-prom"}}

  This means:
    PrometheusRule has label release=kube-prom  → Operator loads it ✓
    PrometheusRule has any other label          → Operator ignores it ✗

  The label name comes from what you named your Helm release when you installed
  kube-prometheus-stack. We named it "kube-prom", so the label must be
  "release: kube-prom".

  How to verify the rule was loaded:
    kubectl get prometheusrule -n helm-lab
    → NAME                    AGE
      flask-cicd-helm2-alerts  5m

  How to verify Prometheus loaded it:
    http://localhost:9090/alerts  → should show FlaskAppDown as INACTIVE

--------------------------------------------------------------------------------
5. WHY absent() INSTEAD OF == 0
--------------------------------------------------------------------------------

  WRONG expression: up{job="flask-cicd-helm2"} == 0
  RIGHT expression: absent(up{namespace="helm-lab"}) or up{namespace="helm-lab"} == 0

  Reason:
    When Flask pods are running:
      Prometheus scrapes them → creates metric: up{namespace="helm-lab"} = 1

    When replicas=0, pods are GONE:
      Prometheus has nothing to scrape → metric DISAPPEARS entirely
      It is NOT set to 0. It just does not exist.

    So "up == 0" never matches because there is no series to compare.

    absent() returns a result when the metric does NOT exist.
    When pods=0 → absent() fires → alert goes PENDING → FIRING ✓

    Also note: job label mismatch was another issue. The ServiceMonitor was
    named "flask-app-monitor" so the scraped job label was "flask-app-monitor"
    not "flask-cicd-helm2". Using namespace="helm-lab" is more reliable.

--------------------------------------------------------------------------------
6. ALERTMANAGER
--------------------------------------------------------------------------------

  What it does:
    - Receives fired alerts from Prometheus
    - Groups similar alerts (deduplication)
    - Routes alerts to receivers: Slack, email, PagerDuty, webhook
    - Supports silences (mute alerts during maintenance)
    - Supports inhibition rules (if major alert fires, suppress minor ones)

  In our setup:
    - alertmanager.enabled=false (disabled to save local RAM)
    - In production: enable it and configure Slack/email receivers

  Port forward to access UI (when enabled):
    kubectl port-forward -n monitoring svc/kube-prom-kube-prometheus-alertmanager 9093:9093
    → http://localhost:9093

  Why we disabled it locally:
    It uses ~150MB RAM. Our Docker Desktop K8s was already near resource limit
    running Prometheus + Grafana + kube-state-metrics + Flask + MariaDB + NGINX.
    Adding Alertmanager caused the cluster to crash. Disabled to keep stable.

--------------------------------------------------------------------------------
7. HOW TO MAKE THE ALERT FIRE (DEMO STEPS)
--------------------------------------------------------------------------------

  Step 1: Port forward Prometheus
    kubectl port-forward -n monitoring svc/kube-prom-kube-prometheus-prometheus 9090:9090

  Step 2: Open http://localhost:9090/alerts → see FlaskAppDown as INACTIVE

  Step 3: Scale Flask to 0 replicas
    kubectl scale deployment flask-cicd-helm2 --replicas=0 -n helm-lab

  Step 4: Wait 90 seconds, refresh browser
    → INACTIVE → PENDING (~30s) → FIRING (~90s)

  Step 5: Bring app back
    kubectl scale deployment flask-cicd-helm2 --replicas=2 -n helm-lab
    → Alert goes back to INACTIVE after a few minutes

10. INTERVIEW Q&A
--------------------------------------------------------------------------------

  Q: What is a PrometheusRule?
  A: A Kubernetes Custom Resource Definition (CRD) added by Prometheus Operator.
     It defines alert conditions in PromQL. Prometheus evaluates the expression
     every 30 seconds. When true for the configured duration, the alert fires.
     In my project I created a FlaskAppDown rule that fires when the Flask app
     has no running pods.

  Q: How does Prometheus Operator discover PrometheusRule objects?
  A: Via label matching. The Operator's ruleSelector defines which labels to look
     for. In my setup it looks for "release: kube-prom" (matching the Helm release
     name). Rules without this label are ignored completely.

  Q: What are the three alert states in Prometheus?
  A: INACTIVE: condition is false, no problem.
     PENDING: condition became true, waiting for the "for:" duration to complete.
     FIRING: condition stayed true for the full duration, alert is active.

  Q: What is Alertmanager?
  A: A component that receives fired alerts from Prometheus and routes them to
     notification channels like Slack, email, or PagerDuty. It also handles
     grouping (deduplication), silences (mute during maintenance), and inhibition
     (suppress minor alerts if a major one is firing).

  Q: Why did you use absent() instead of checking up == 0?
  A: When all pods are gone (replicas=0), Prometheus has no targets to scrape.
     The "up" metric disappears entirely -- it's not set to 0, it just doesn't
     exist. The expression "up == 0" would never match because there's no series
     to compare. absent() specifically detects when a metric is missing, which is
     exactly the "app completely gone" scenario.

  Q: What is PromQL?
  A: Prometheus Query Language. Used to query and aggregate time-series metrics.
     Example: absent(up{namespace="helm-lab"}) means "is there NO 'up' metric
     in the helm-lab namespace?" -- returns a result when nothing is scraped there.

  Q: How would you set up Slack notifications from Prometheus?
  A: Configure Alertmanager with a Slack receiver in its config:
     - Define a route matching alert labels (e.g., severity=critical)
     - Define a receiver with the Slack webhook URL and channel
     - Alertmanager sends a formatted message to Slack when alert fires.
     In kube-prometheus-stack this is done via Helm values under alertmanager.config.

                    STAGE 15: LOGGING (Loki + Promtail + Grafana)
================================================================================

WHAT THIS STAGE DOES:
  Our Flask app prints logs to stdout. With "kubectl logs" we can see them
  one pod at a time. The problem: when a pod crashes, logs are gone. When
  there are 10 pods, we can't manually check each one.

  This stage installs a CENTRAL log collection system:
    Promtail collects → Loki stores → Grafana shows all logs in one place.

--------------------------------------------------------------------------------
1. THE THREE PLAYERS (easy way)
--------------------------------------------------------------------------------

  Promtail (LOG COLLECTOR)
    - Runs as a DaemonSet (one pod per node).
    - Reads pod log files from /var/log/pods/...
    - Adds K8s labels (namespace, pod, app) to each line.
    - Ships logs to Loki over HTTP.

  Loki (LOG STORAGE + QUERY API)
    - Runs as a StatefulSet (port 3100).
    - Receives logs from Promtail.
    - Indexes ONLY the labels (not the full log text), so it's lightweight.
    - Provides an HTTP query API for LogQL.

  Grafana (VISUALIZER - already installed in Stage 12)
    - Asks Loki for logs over HTTP.
    - Shows them in the Explore tab.
    - We add Loki as a "datasource" in Grafana.

  Flow:
    Flask pod → stdout → /var/log/pods file → Promtail reads → ships to Loki
    User in Grafana Explore → types LogQL → Grafana asks Loki → shows logs

--------------------------------------------------------------------------------
2. INSTALL (ONE HELM COMMAND)
--------------------------------------------------------------------------------

  helm repo add grafana https://grafana.github.io/helm-charts
  helm repo update
  helm install loki grafana/loki-stack -n monitoring \
    --set grafana.enabled=false \      ← we already have Grafana
    --set prometheus.enabled=false     ← we already have Prometheus

  This creates:
    StatefulSet: loki-0
    DaemonSet:   loki-promtail-<random>
    Service:     loki (ClusterIP, port 3100)
    ConfigMap:   loki-loki-stack (datasource definition for Grafana)

--------------------------------------------------------------------------------
3. THE MAGIC: HOW GRAFANA AUTO-DETECTED LOKI
--------------------------------------------------------------------------------

  We did NOT manually add Loki as a datasource in the Grafana UI.
  It just appeared. Here is exactly why.

  The Grafana pod from kube-prometheus-stack actually has 3 containers:
    grafana                    ← the main app
    grafana-sc-dashboard       ← sidecar watching for dashboard ConfigMaps
    grafana-sc-datasources     ← sidecar watching for datasource ConfigMaps

  The DATASOURCES sidecar watches for ConfigMaps with this label:
    grafana_datasource: "1"

  The DASHBOARDS sidecar watches for a different label:
    grafana_dashboard: "1"

  When you installed loki-stack, it created a ConfigMap called
  "loki-loki-stack" with label grafana_datasource: "1" and this content:

    apiVersion: 1
    datasources:
    - name: Loki
      type: loki
      url: "http://loki:3100"
      isDefault: false

  The sidecar saw the label → loaded the file into Grafana → Loki appeared
  in the UI automatically.

  Same pattern is used for our Flask dashboard:
    helm/flask-cicd-app/templates/grafana-dashboard-configmap.yaml
    has label: grafana_dashboard: "1"
    The dashboard sidecar picks it up the same way.

  Two sidecars, two labels, two different jobs. Same pattern.

--------------------------------------------------------------------------------
4. THE CRASHLOOP STORY (Real Issue Faced)
--------------------------------------------------------------------------------

  After installing Loki, Grafana went into CrashLoopBackOff.

  Logs showed:
    Datasource provisioning error: datasource.yaml config is invalid.
    Only one datasource per organization can be marked as default

  Root cause: TWO datasource ConfigMaps both had isDefault: true
    1. kube-prom-kube-prometheus-grafana-datasource  → Prometheus, default=true
    2. loki-loki-stack                               → Loki, default=true

  Each Helm chart sets itself as default by default (it doesn't know other
  charts exist). Grafana refuses to start if it sees two defaults.

  Fix: change the Loki ConfigMap to isDefault: false
       (Prometheus stays default because we query metrics more often).

  Result: Grafana booted cleanly. Both datasources still available.
  Loki is just not auto-selected - we pick it manually in Explore.

  Important takeaway:
    isDefault: true   = pre-selected in dropdowns when you open them
    isDefault: false  = still available, you just have to pick it manually
    The datasource is loaded either way - the label "grafana_datasource: 1"
    is what makes it appear. isDefault only controls which one is auto-shown.

--------------------------------------------------------------------------------
5. WHO TALKS TO WHO (CRITICAL UNDERSTANDING)
--------------------------------------------------------------------------------

  Common confusion: people think Prometheus pushes data to Grafana.
  TRUTH: Grafana is a CLIENT. It asks others for data over HTTP.

  Grafana → asks Prometheus → "give me CPU values"
  Grafana → asks Loki       → "give me log lines matching this query"
  Grafana → asks MariaDB    → "give me table rows"

  Loki and Prometheus DO NOT talk to each other. They are completely
  independent systems that share only one thing: Grafana queries both.

  Metrics path:
    Flask /metrics endpoint → Prometheus scrapes → stores → Grafana queries

  Logs path:
    Flask stdout → /var/log/pods file → Promtail ships → Loki stores
    → Grafana queries

  Two parallel pipelines. One visualization layer (Grafana) on top.

--------------------------------------------------------------------------------
6. LOGQL (THE QUERY LANGUAGE)
--------------------------------------------------------------------------------

  Same syntax style as PromQL. Two parts:
    {label_selector}  |= "text filter"   (optional filter)

  Examples:
    {namespace="helm-lab"}                   → all logs from Flask namespace
    {namespace="helm-lab", app="flask-cicd-app"}  → only Flask app logs
    {namespace="helm-lab"} |= "/health"      → only lines with /health
    {namespace="helm-lab"} |= "ERROR"        → only error lines
    {namespace="helm-lab"} != "/metrics"     → exclude metrics scrape lines

  Operators:
    |=  → contains
    !=  → does not contain
    |~  → regex match
    !~  → regex no match

--------------------------------------------------------------------------------
7. LOKI VS ELK STACK

  Visualization     Grafana                    Kibana

  We picked Loki because:
    - Same Grafana we already use
    - Light on resources (Docker Desktop has limits)

10. INTERVIEW Q&A

  Q: How is logging set up in your project?
  A: I use Loki for centralized log storage and Promtail as the log collector.
     Promtail runs as a DaemonSet, reads pod log files from /var/log/pods,
     adds Kubernetes labels (namespace, pod, app), and ships to Loki on

     lighter than full-text indexing like Elasticsearch. I view logs in
     Grafana's Explore tab using LogQL queries.

  Q: How did Grafana detect Loki without manual setup?
  A: The kube-prometheus-stack Grafana pod has a sidecar container that
     watches the K8s API for ConfigMaps with the label grafana_datasource: "1".
     The loki-stack Helm chart automatically creates such a ConfigMap with
     the Loki connection URL. The sidecar loads it into Grafana on the fly -
     no Grafana restart, no manual UI clicks.

  Q: Difference between Loki and ELK?
  A: Loki indexes only labels, ELK indexes full log text. Loki uses LogQL
     (simple), ELK uses Lucene/KQL (more powerful but heavier). Loki is
     much lighter on RAM and CPU. ELK is better for advanced search at scale.
     For K8s cloud-native setups, Loki is the standard choice.

  Q: What is Promtail and where does it run?
  A: Promtail is a log shipper. It runs as a DaemonSet so there's exactly
     one Promtail pod per cluster node. It tails the pod log files on that
     node, applies labels, and POSTs the lines to Loki's push API.

  Q: Have you debugged a Grafana issue?
  A: Yes, my Grafana crashed after I added Loki. Logs showed two datasources
     both marked as default - the kube-prometheus-stack chart had marked
     Prometheus as default, and the loki-stack chart had marked Loki as
     default too. Grafana refuses to start with two defaults. I set Loki's
     isDefault to false, since Prometheus is queried more often. Grafana
     restarted cleanly. Both datasources still queryable; only the default
     selection changed.

  Q: Sample LogQL query?
  A: {namespace="helm-lab", app="flask-cicd-app"} |= "ERROR"
     Selects logs from my Flask app in the helm-lab namespace,
     filtered to lines containing "ERROR".

                    STAGE 16: RBAC & SECURITY (ServiceAccount + Role + RoleBinding)
================================================================================

WHAT THIS STAGE DOES:
  Until now our Flask pod used the namespace's "default" ServiceAccount which
  has ZERO permissions. That's actually fine because Flask doesn't call the
  K8s API, but it's bad practice. RBAC is THE most asked security topic in
  K8s interviews. This stage gives the Flask pod its own identity with
  explicit, minimal permissions -- following the principle of least privilege.

--------------------------------------------------------------------------------
1. THE CORE CONCEPT (EASY WAY)
--------------------------------------------------------------------------------

  RBAC = Role-Based Access Control
  It answers: WHO can do WHAT, WHERE?

  WHO        - User (humans) OR ServiceAccount (pods/code)
  WHAT       - Verbs (get, list, create, delete) on Resources (pods, secrets)
  WHERE      - One namespace (Role) OR whole cluster (ClusterRole)
  LINK       - RoleBinding connects WHO to WHAT

  Without a binding, K8s denies everything (deny-by-default).

--------------------------------------------------------------------------------
2. THE 4 OBJECTS WORKING TOGETHER
--------------------------------------------------------------------------------

  ┌─────────────────┐
  │ ServiceAccount  │  ← WHO am I? (identity for code/pods)
  │ flask-cicd-     │
  │ helm2-sa        │
  └────────┬────────┘
           │ subjects (left)
           │
  ┌────────┴────────┐
  │   RoleBinding   │  ← THE LINK (binds WHO to WHAT)
  │ ...binding      │
  └────────┬────────┘
           │ roleRef (right)
           │
  ┌────────┴────────┐
  │      Role       │  ← WHAT can I do?
  │ ...role         │
  │ - configmaps    │
  │   get, list     │
  └─────────────────┘

  Namespace-scoped: Role + RoleBinding
  Cluster-scoped:   ClusterRole + ClusterRoleBinding

--------------------------------------------------------------------------------
3. FILES TOUCHED (3 FILES IN THIS STAGE)
--------------------------------------------------------------------------------

  1. helm/flask-cicd-app/templates/rbac.yaml   ← NEW: SA + Role + RoleBinding
  2. helm/flask-cicd-app/values.yaml           ← Set serviceAccount.name + rbac.create
  3. Jenkinsfile (main branch)                  ← Added --set rbac.create=true

--------------------------------------------------------------------------------
4. rbac.yaml (THE NEW TEMPLATE)
--------------------------------------------------------------------------------

  {{- if .Values.rbac.create }}
  apiVersion: v1
  kind: ServiceAccount
  metadata:
    name: {{ include "flask-cicd-app.fullname" . }}-sa
    namespace: {{ .Release.Namespace }}
  ---
  apiVersion: rbac.authorization.k8s.io/v1
  kind: Role
  metadata:
    name: {{ include "flask-cicd-app.fullname" . }}-role
    namespace: {{ .Release.Namespace }}
  rules:
    - apiGroups: [""]
      resources: ["configmaps"]
      verbs: ["get", "list"]
  ---
  apiVersion: rbac.authorization.k8s.io/v1
  kind: RoleBinding
  metadata:
    name: {{ include "flask-cicd-app.fullname" . }}-binding
    namespace: {{ .Release.Namespace }}
  subjects:
    - kind: ServiceAccount
      name: {{ include "flask-cicd-app.fullname" . }}-sa
      namespace: {{ .Release.Namespace }}
  roleRef:
    kind: Role
    name: {{ include "flask-cicd-app.fullname" . }}-role
    apiGroup: rbac.authorization.k8s.io
  {{- end }}

  All 3 objects in one file. All gated by rbac.create flag.

--------------------------------------------------------------------------------
5. values.yaml CHANGES (WHY EACH ONE)
--------------------------------------------------------------------------------

  serviceAccount:
    create: false                       # chart's built-in SA template stays OFF
                                        # (we create SA in rbac.yaml instead)
    name: "flask-cicd-helm2-sa"         # MUST match what rbac.yaml creates
                                        # so deployment.yaml helper picks it up

  rbac:
    create: true                        # turn ON our rbac.yaml template

  Three knobs, three jobs:
    serviceAccount.create  - "Should chart auto-create a basic SA?"  (NO)
    serviceAccount.name    - "What name should deployment look for?" (our SA)
    rbac.create            - "Should our rbac.yaml run?"             (YES)

--------------------------------------------------------------------------------
6. WHY KEEP serviceAccount.create: false?
--------------------------------------------------------------------------------

  There are TWO places an SA could be created in a Helm chart:

    Option A: Chart's built-in serviceaccount.yaml template
              (controlled by serviceAccount.create)
    Option B: Our custom rbac.yaml template
              (controlled by rbac.create)

  If BOTH were true → both would try to create the SAME SA → Helm error.

  We picked Option B because:
    - All RBAC pieces (SA + Role + Binding) live in ONE file
    - Easier to read, easier to enable/disable as a unit
    - Cleaner architectural separation

11. THE 0-BYTES BUG (REAL DEBUG STORY)
--------------------------------------------------------------------------------

  Initial attempt failed silently. Helm reported "upgrade complete" but no
  RBAC objects appeared in the cluster. Investigation:

    helm template flask-cicd-helm2 helm/flask-cicd-app -s templates/rbac.yaml
    → Error: could not find template templates/rbac.yaml in chart

    Get-ChildItem templates/ | Format-Table Name, Length
    → rbac.yaml  0   <-- FILE WAS 0 BYTES

  Root cause: the editor showed content from memory but the file was never
  saved to disk. Once written properly (38 lines), Helm rendered it correctly.

  Lesson: when Helm silently skips a template, always check Length on disk,
  not just what the editor shows.

12. VERBS AND RESOURCES CHEAT SHEET
--------------------------------------------------------------------------------

  Verbs:
    get             read one specific object
    list            read many objects
    watch           subscribe to changes (used by controllers)
    create          make new
    update          replace entirely
    patch           modify partially
    delete          remove
    deletecollection  bulk delete
    *               all (avoid; violates least privilege)

  Common resources and their apiGroups:
    pods, services, secrets, configmaps, nodes    →  apiGroups: [""]
    deployments, replicasets, statefulsets        →  apiGroups: ["apps"]
    ingresses, networkpolicies                    →  apiGroups: ["networking.k8s.io"]
    roles, rolebindings                           →  apiGroups: ["rbac.authorization.k8s.io"]
    prometheusrules, servicemonitors              →  apiGroups: ["monitoring.coreos.com"]

  Find any resource's apiGroup:
    kubectl api-resources

13. ROLE vs CLUSTERROLE (QUICK REFERENCE)
--------------------------------------------------------------------------------

                Role                       ClusterRole
  Scope         one namespace              entire cluster
  Pair with     RoleBinding                ClusterRoleBinding
  Used by       app teams (per namespace)  Prometheus, NGINX Ingress, etc.

  CURVEBALL: RoleBinding CAN bind a ClusterRole. This grants the ClusterRole's
  permissions but ONLY within the binding's namespace -- useful for reusing
  built-in ClusterRoles like "view" / "edit" / "admin" per namespace.

  RBAC is ONE slice of K8s security. The big picture:

  NetworkPolicy           Pod-to-pod firewall (e.g., only Flask → MariaDB)
  Pod Security Standards  Restrict pods (no root, no host network)
  Secrets at rest         K8s Secrets are base64, not encrypted by default.

  Image scanning          Trivy/Snyk scan images for CVEs in CI/CD
  Admission controllers   OPA / Gatekeeper - block bad YAML before deploy
  Audit logs              K8s logs every API call (SOC 2, ISO 27001 needs)

15. INTERVIEW Q&A
--------------------------------------------------------------------------------

  Q: What is RBAC and why does Kubernetes use it?
  A: Role-Based Access Control. It answers WHO can do WHAT, WHERE. Without
     RBAC, anyone (or any pod) could delete production resources. K8s defaults
     to deny-everything and only allows actions explicitly granted via Roles
     and RoleBindings -- the principle of least privilege.

  Q: Difference between Role and ClusterRole?
  A: Role is scoped to one namespace. ClusterRole is cluster-wide. Use Role
     for application-level permissions. Use ClusterRole for tools that span
     namespaces like Prometheus or NGINX Ingress Controller. ClusterRoles can
     also be bound by a RoleBinding to grant cluster-role permissions within
     a single namespace.

  Q: What's a ServiceAccount and how does a pod use it?
  A: A ServiceAccount is an identity for pods/code (vs Users which are for
     humans). When a pod starts, K8s mounts the SA's JWT token at
     /var/run/secrets/kubernetes.io/serviceaccount/token. The pod sends this
     token to the K8s API; the API server validates it and looks up
     RoleBindings to decide what's allowed.

  Q: What does the "default" ServiceAccount allow?
  A: In modern K8s, nothing. Every namespace gets a "default" SA but it has
     no RoleBindings, so any API call from it returns 403 Forbidden. Production
     best practice: always use a named SA per workload.

  Q: How do you debug "Forbidden" errors?
  A: kubectl auth can-i <verb> <resource> --as=system:serviceaccount:<ns>:<sa>
     Example: kubectl auth can-i delete pods --as=system:serviceaccount:helm-lab:flask-cicd-helm2-sa
     If it says "no", check the SA's RoleBindings and ensure the Role grants
     that verb on that resource.

  Q: Walk me through your RBAC setup in the Flask project.
  A: I bundled all RBAC pieces in one Helm template (rbac.yaml): a ServiceAccount,
     a Role allowing only get/list on configmaps, and a RoleBinding linking them.
     It's gated by rbac.create flag in values.yaml. The deployment.yaml's
     serviceAccountName field uses a Helm helper that resolves to the same
     name (flask-cicd-helm2-sa). I verified with kubectl auth can-i -- the SA
     can list configmaps (granted) but cannot delete pods (not granted).

  Q: What's the principle of least privilege?
  A: Grant the minimum permissions an identity needs to do its job and nothing
     more. Prefer Role over ClusterRole. Avoid wildcards (*) in verbs and
     resources. Use specific resource names where possible.
